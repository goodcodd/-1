# Лабораторна робота 3: Пошук та фільтрація в React

## Фрагменти коду

### Логіка роботи пошуку

Пошук реалізовано через керований компонент `SearchBar`, який отримує значення та функцію оновлення через props:

```jsx
// PostsPage.jsx
const [searchTerm, setSearchTerm] = useState('');

// Передача стану та функції оновлення в SearchBar
<SearchBar 
  searchTerm={searchTerm} 
  onSearchChange={setSearchTerm} 
/>

// SearchBar.jsx - керований компонент
const SearchBar = ({ searchTerm, onSearchChange }) => {
  return (
    <div className={styles.searchWrapper}>
      <input
        type="text"
        placeholder="Пошук постів за автором або вмістом..."
        value={searchTerm}  // Значення контролюється ззовні
        onChange={(e) => onSearchChange(e.target.value)}  // Оновлення через callback
        className={styles.searchInput}
      />
      {searchTerm && (
        <button
          onClick={() => onSearchChange('')}  // Очищення пошуку
          className={styles.clearButton}
        >
          ×
        </button>
      )}
    </div>
  );
};
```

**Як працює пошук:**
1. Стан `searchTerm` зберігається в батьківському компоненті `PostsPage`
2. При введенні тексту викликається `onSearchChange`, який оновлює `searchTerm`
3. Оновлення стану викликає перерендер компонента
4. Фільтрація відбувається автоматично через `useMemo`

### Логіка фільтрації

Фільтрація поєднує пошук за текстом та фільтрацію за категорією:

```jsx
// PostsPage.jsx
const [searchTerm, setSearchTerm] = useState('');
const [activeCategory, setActiveCategory] = useState('All');

// Логіка фільтрації з використанням useMemo для оптимізації
const filteredPosts = useMemo(() => {
  return postsData.filter(post => {
    // Фільтрація за пошуковим запитом
    const matchesSearch = 
      post.content.toLowerCase().includes(searchTerm.toLowerCase()) ||
      post.author.toLowerCase().includes(searchTerm.toLowerCase());
    
    // Фільтрація за категорією
    const matchesCategory = activeCategory === 'All' || post.category === activeCategory;
    
    // Пост проходить фільтр, якщо відповідає обом умовам
    return matchesSearch && matchesCategory;
  });
}, [searchTerm, activeCategory]);  // Перерахунок тільки при зміні цих значень
```

**Особливості реалізації:**
- Використовується `useMemo` для оптимізації - фільтрація виконується тільки при зміні `searchTerm` або `activeCategory`
- Пошук нечутливий до регістру (`.toLowerCase()`)
- Пошук працює одночасно за автором та вмістом поста
- Фільтрація за категорією: якщо вибрано "All", показуються всі пости
- Умови поєднуються через логічне І (`&&`)

### Обробка "порожнього результату" (Empty State)

Коли фільтрація не знаходить жодних результатів, відображається спеціальний компонент:

```jsx
// PostsPage.jsx
<div className={styles.feed}>
  {filteredPosts.length > 0 ? (
    // Відображення знайдених постів
    filteredPosts.map((post) => (
      <Post
        key={post.id}
        author={post.author}
        content={post.content}
        date={post.date}
        avatar={post.avatar}
        initialLikes={post.likes}
      />
    ))
  ) : (
    // Відображення Empty State
    <EmptyState />
  )}
</div>

// EmptyState.jsx
const EmptyState = ({ message = "Нічого не знайдено за вашим запитом." }) => {
  return (
    <div className={styles.emptyState}>
      <div className={styles.icon}>🔍</div>
      <p className={styles.message}>{message}</p>
      <p className={styles.hint}>
        Спробуйте змінити параметри пошуку або вибрати іншу категорію
      </p>
    </div>
  );
};
```

**Як працює обробка порожнього результату:**
1. Перевірка кількості: `filteredPosts.length > 0`
2. Умовний рендеринг через тернарний оператор
3. Якщо результатів немає - відображається `EmptyState` з:
   - Візуальною іконкою
   - Повідомленням для користувача
   - Підказкою, що робити далі
4. Якщо є результати - відображається список постів

**Переваги такого підходу:**
- Покращує UX - користувач розуміє, що пошук не дав результатів
- Надає корисні підказки
- Візуально відрізняється від списку постів
- Можна налаштувати повідомлення для різних сценаріїв

## Контрольні запитання

### 1. Що таке "підняття стану" (lifting state up) і чому ми використовуємо його для `SearchBar`?

**"Підняття стану" (Lifting State Up)** — це патерн у React, коли стан компонента переміщується до найближчого спільного предка компонентів, які його використовують.

**Як це працює:**
- Замість зберігання стану в дочірньому компоненті (`SearchBar`), стан зберігається в батьківському (`PostsPage`)
- Дочірній компонент отримує значення та функцію оновлення через props
- Зміни стану відбуваються через callback-функції, передані з батьківського компонента

**Чому використовуємо для `SearchBar`:**
1. **Спільне використання стану**: Значення пошуку потрібне не тільки `SearchBar`, але й для фільтрації в `PostsPage`
2. **Одно джерело правди**: Стан зберігається в одному місці, що уникнення розсинхронізації
3. **Контроль зверху**: Батьківський компонент контролює, як використовується значення пошуку
4. **Повторне використання**: `SearchBar` залишається "чистим" компонентом без внутрішнього стану, його можна використовувати в різних місцях

**Приклад:**
```jsx
// Погано - стан в дочірньому компоненті
const SearchBar = () => {
  const [searchTerm, setSearchTerm] = useState('');
  // Як передати searchTerm в PostsPage? Неможливо!
};

// Добре - стан піднято в батьківський компонент
const PostsPage = () => {
  const [searchTerm, setSearchTerm] = useState('');  // Стан тут
  return <SearchBar searchTerm={searchTerm} onSearchChange={setSearchTerm} />;
};
```

### 2. Поясніть асинхронну природу оновлення стану в `useState`.

**Асинхронність оновлення стану** означає, що виклик `setState` не оновлює значення стану миттєво. React планує оновлення та виконує його під час наступного рендеру.

**Ключові моменти:**

1. **Батчинг (Batching)**:
   ```jsx
   const [count, setCount] = useState(0);
   
   const handleClick = () => {
     setCount(count + 1);  // count все ще 0
     setCount(count + 1);  // count все ще 0
     // Після рендеру count буде 1, а не 2!
   };
   ```
   React об'єднує кілька оновлень стану в один рендер для оптимізації.

2. **Функціональне оновлення**:
   ```jsx
   const handleClick = () => {
     setCount(prevCount => prevCount + 1);  // prevCount = 0
     setCount(prevCount => prevCount + 1);  // prevCount = 1
     // Після рендеру count буде 2!
   };
   ```
   Використання функції гарантує, що ми працюємо з актуальним значенням.

3. **Після оновлення стану**:
   ```jsx
   const [searchTerm, setSearchTerm] = useState('');
   
   const handleChange = (value) => {
     setSearchTerm(value);
     console.log(searchTerm);  // Все ще старе значення!
     // Правильне значення буде доступне після наступного рендеру
   };
   ```

**Наслідки для нашого коду:**
- Коли користувач вводить текст, `setSearchTerm` викликається, але `searchTerm` оновлюється тільки після рендеру
- `useMemo` автоматично перераховує `filteredPosts` після оновлення `searchTerm`
- Це забезпечує синхронізацію між вводом користувача та результатами фільтрації

### 3. Чому для фільтрації ми створюємо нову змінну `filteredPosts`, а не змінюємо оригінальний масив `postsData` у стані?

**Причини створення нової змінної:**

1. **Іммутабельність (Immutability)**:
   - React працює з принципом іммутабельності - не змінюємо існуючі дані, а створюємо нові
   - Це допомагає React визначати зміни та оптимізувати рендеринг

2. **Збереження оригінальних даних**:
   ```jsx
   // Погано - втрачаємо оригінальні дані
   const filteredPosts = postsData.filter(...);
   postsData = filteredPosts;  // Тепер postsData змінений!
   
   // Добре - оригінальні дані залишаються незмінними
   const filteredPosts = postsData.filter(...);
   // postsData залишається незмінним, можемо використати знову
   ```

3. **Множинне використання**:
   - `postsData` може використовуватися в різних місцях
   - Якщо ми змінимо його, це вплине на всі місця використання
   - Створення нової змінної ізольовано від інших частин коду

4. **Оптимізація з `useMemo`**:
   ```jsx
   const filteredPosts = useMemo(() => {
     return postsData.filter(...);
   }, [searchTerm, activeCategory]);
   ```
   - `useMemo` кешує результат і перераховує тільки при зміні залежностей
   - Якщо б ми змінювали `postsData`, це б зламало механізм кешування

5. **Відстеження змін React**:
   - React порівнює посилання на об'єкти для визначення змін
   - Якщо ми мутуємо масив, посилання залишається тим самим, React не помітить зміни
   - Створення нового масиву створює нове посилання, React правильно визначає зміни

**Приклад проблеми з мутацією:**
```jsx
// Погано
const [posts, setPosts] = useState(postsData);
posts.filter(...);  // Мутуємо масив
// React може не помітити зміни!

// Добре
const filteredPosts = postsData.filter(...);  // Новий масив
// React точно помітить зміни
```

### 4. У чому перевага використання керованих компонентів над некерованими при реалізації пошуку?

**Керовані компоненти (Controlled Components)** — це компоненти, значення яких контролюється React через `value` prop та `onChange` handler.

**Некеровані компоненти (Uncontrolled Components)** — це компоненти, які зберігають свій стан у DOM (через `ref`).

**Переваги керованих компонентів:**

1. **Єдине джерело правди**:
   ```jsx
   // Керований - стан в React
   <input value={searchTerm} onChange={(e) => setSearchTerm(e.target.value)} />
   
   // Некерований - стан в DOM
   <input ref={inputRef} />
   const value = inputRef.current.value;  // Потрібно читати з DOM
   ```
   У керованому компоненті стан завжди в React, легко отримати значення.

2. **Валідація та обробка в реальному часі**:
   ```jsx
   // Керований - можемо обробляти під час введення
   <input 
     value={searchTerm}
     onChange={(e) => {
       const value = e.target.value;
       // Можемо валідувати, фільтрувати, трансформувати
       setSearchTerm(value.toLowerCase());  // Наприклад, перевести в нижній регістр
     }}
   />
   ```

3. **Програмний контроль**:
   ```jsx
   // Керований - можемо змінити значення програмно
   const clearSearch = () => {
     setSearchTerm('');  // Легко очистити
   };
   
   // Некерований - потрібно маніпулювати DOM
   const clearSearch = () => {
     inputRef.current.value = '';  // Пряма маніпуляція DOM
   };
   ```

4. **Синхронізація з іншими компонентами**:
   ```jsx
   // Керований - легко синхронізувати
   <SearchBar searchTerm={searchTerm} onSearchChange={setSearchTerm} />
   <button onClick={() => setSearchTerm('')}>Очистити</button>
   // Обидва компоненти працюють з одним станом
   ```

5. **Тестування**:
   - Керовані компоненти легше тестувати - можна передати значення та перевірити зміни
   - Некеровані потребують мокування DOM

6. **Інтеграція з формами**:
   - Легко інтегрувати з бібліотеками форм (Formik, React Hook Form)
   - Легше реалізувати складну логіку валідації

**Недоліки керованих компонентів:**
- Трохи більше коду (потрібно передавати `value` та `onChange`)
- Можливі проблеми з продуктивністю при великій кількості полів (але це рідкість)

**Висновок:**
Для пошуку керовані компоненти ідеальні, оскільки:
- Значення пошуку потрібне для фільтрації в реальному часі
- Потрібна можливість програмно очищати поле
- Потрібна синхронізація з іншими елементами UI (наприклад, кнопка очищення)
- Легше тестувати та підтримувати
