# Лабораторна робота 7: Робота з URL-параметрами (React Router)

## Контрольні запитання

### 1. Різниця між Path Parameters та Query Parameters. Коли що використовувати?

**Path Parameters** (параметри шляху), наприклад `/users/42`:
- Частина **шляху** URL; ідентифікують **конкретний ресурс** (один користувач, один пост).
- Використовуються, коли значення **обов’язкове** і визначає, **яку сторінку/ресурс** показувати (наприклад, ID поста, ID користувача).
- Приклад: `/feed/42` — пост з ID 42.

**Query Parameters** (параметри запиту), наприклад `/users?sort=asc&page=2`:
- Розташовані **після `?`**; описують **фільтри, сортування, пошук, пагінацію** тощо.
- Можуть бути **опційними** і комбінуватися (кілька параметрів одночасно).
- Приклад: `/feed?query=react&category=Tech&sort=desc`.

**Висновок:** Path — для ідентифікації ресурсу (що показувати); Query — для стану перегляду (як фільтрувати, сортувати, шукати).

### 2. Що робить useParams() і який синтаксис у &lt;Route path="..."&gt; потрібен?

**useParams()** — хук React Router, який повертає об’єкт з **параметрами шляху** поточного маршруту. Назви ключів відповідають іменам, заданим у `path`.

**Синтаксис у Route:** у `path` динамічний сегмент позначається **двокрапкою та іменем**:
- `path="/post/:postId"` — параметр `postId` (наприклад, для `/post/42` буде `{ postId: "42" }`).
- `path="/users/:userId/posts/:postId"` — два параметри: `userId`, `postId`.

У компоненті: `const { postId } = useParams();` — отримуємо значення з URL. Без `:ім'я` у `path` хук не поверне цей параметр.

### 3. Чому зберігання пошукового рядка в URL (useSearchParams) краще, ніж useState? Переваги для користувача

**Архітектурні переваги URL-стану:**
- Стан пошуку/фільтрів **зберігається в адресному рядку** і **синхронізується** з історією браузера.

**Переваги для кінцевого користувача:**
1. **Посилання можна надіслати** — інший користувач відкриє той самий результат пошуку або фільтрації.
2. **Оновлення сторінки (F5)** не скидає пошук і фільтри — стан «виживає».
3. **Кнопка «Назад»** повертає до попередніх параметрів пошуку, як до попередньої сторінки.
4. **Закладки** — можна зберегти посилання з потрібним пошуком/фільтром і відкрити пізніше.

При зберіганні лише в `useState` жодна з цих можливостей недоступна: після перезавантаження або відкриття посилання стан губиться.

### 4. Чи відбувається фільтрація при кожному рендері, якщо filter() викликати прямо перед JSX? Вплив на продуктивність і оптимізація

Так. Якщо писати щось на кшталт:
```js
const filtered = items.filter(...);
return <div>{filtered.map(...)}</div>;
```
то **фільтрація виконується при кожному рендері** компонента (кожна зміна стану або батьківського рендеру викликає перерахунок).

**Вплив на продуктивність:** для дуже великих масивів (наприклад, 10 000+ елементів) це дає зайві обчислення і може сповільнити інтерфейс.

**Оптимізація:** використовувати хук **useMemo**:
```js
const filteredPosts = useMemo(() => {
  return items.filter(...);
}, [items, searchQuery, activeCategory]); // перерахунок лише при зміні залежностей
```
Тоді фільтрація виконується лише коли змінюються залежності (наприклад, масив даних або параметри пошуку), а не при кожному рендері.

### 5. В яких об’єктах Web API зберігаються параметри після `?`? Як React Router це спрощує?

У звичайному JavaScript параметри після `?` (query string) представлені в Web API так:
- **window.location.search** — рядок усього query string (наприклад, `"?query=react&sort=desc"`).
- **URLSearchParams** — стандартний клас для парсингу та зміни цих параметрів:
  - `new URLSearchParams(window.location.search)` — створити об’єкт з поточного URL;
  - `.get('query')`, `.set('query', 'react')`, `.toString()` тощо.

Треба вручну парсити рядок, оновлювати параметри, викликати `history.pushState` або змінювати `window.location`, щоб URL оновлювався і не перезавантажував сторінку.

**React Router спрощує:** хук **useSearchParams()** повертає `[searchParams, setSearchParams]`:
- **searchParams** — об’єкт, подібний до `URLSearchParams` (методи `.get()`, `.has()` тощо).
- **setSearchParams()** — оновлює query string в URL і **перерендерює** компоненти без перезавантаження сторінки.

Тобто не потрібно вручну працювати з `window.location` та історією браузера — стан URL синхронізується з React, і маршрутизація залишається в межах SPA.
