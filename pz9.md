# Практична робота №9. Контрольні запитання — відповіді

## 1. Принципова різниця в області застосування між `useMemo` та `React.memo`

**`useMemo`** — це **хук**, який кешує **результат обчислення** (значення), повернутого функцією, між рендерами **одного й того ж компонента**. Він не “зупиняє” рендер самого компонента: компонент усе одно викликається, якщо батько його оновив; `useMemo` лише дозволяє **не повторювати дорогі обчислення** (фільтрація великих масивів, складні розрахунки тощо), якщо залежності не змінились.

**`React.memo`** — це **компонент вищого порядку (HOC)**, який обгортає **функціональний компонент** і каже React: **не викликати повторно функцію компонента**, якщо **props** (поверхнево порівняні) не змінились. Тобто область застосування — **оптимізація рендеру дочірнього компонента** за рахунок пропуску повторного виконання його тіла.

**Коротко:** `useMemo` оптимізує **обчислення всередині** компонента; `React.memo` — **частоту рендерів самого компонента** залежно від props.

## 2. React Profiler: жовтий/червонуватий vs зелений/сірий на Flamegraph

- **Жовтий / більш “гарячий” (червонуватий)** відтінок зазвичай означає **довший час рендеру** цього вузла в обраному коміті: компонент зайняв більше часу на фазі **render** відносно інших. Чим довше (часто ближче до “червоного”), тим **більший внесок** у загальний час кадру — потенційне “вузьке місце”.

- **Зелений** часто асоціюється з **коротшим / “дешевшим”** рендером у межах того ж масштабу кольорової шкали Profiler (швидше відпрацював).

- **Сірий** зазвичай означає, що компонент **не рендерився** (наприклад, “Did not render” / пропущено оновлення) або **не брав участі** в цьому коміті так, як інші вузли — залежно від версії DevTools і режиму перегляду.

**Важливо:** точні відтінки можуть трохи відрізнятися в залежності від версії React DevTools; орієнтир — **довжина смуги (duration)** і **легенда** біля Flamegraph.

## 3. Чому `React.memo` + `onClick={handleClick}` майже завжди потребує `useCallback` у батька (Referential Equality)

`React.memo` порівнює **попередні** та **нові** props **поверхнево** (для примітивів — за значенням, для **об’єктів і функцій** — за **посиланням** у пам’яті).

Якщо в батьківському компоненті при кожному рендері писати:

```jsx
<Child onClick={() => doSomething()} />
```

або

```jsx
const handleClick = () => { ... };
<Child onClick={handleClick} />
```

без `useCallback`, то **function expression** `handleClick` при кожному рендері батька — це **нова функція** (нове посилання). Для `React.memo` це виглядає як **зміна props**, навіть якщо логіка та сама. Тому дочірній компонент **все одно перемальовується**.

**`useCallback`** фіксує **те саме посилання** на функцію між рендерами (поки не зміняться залежності з масиву), тож `onClick` як prop **не “новий”** при кожному рендері батька — і `React.memo` може **реально пропустити** рендер дитини.

Це і є **referential equality**: рівність для функцій у JS — це “чи це той самий об’єкт у пам’яті”, а не “чи однаковий код”.

## 4. Чи є накладні витрати на `useMemo` / `useCallback`? Коли “оптимізація” шкодить

Самі хуки додають **роботу**: для кожного виклику React має **зберігати** попередні значення/залежності і **порівнювати** їх масиви залежностей. Це недорого, але **не нульова** ціна.

**Коли мінусів більше, ніж плюсів (об’єктивний приклад):**

- Обгорнути в `useMemo` **тривіальне** обчислення, наприклад `useMemo(() => a + b, [a, b])` або `useMemo(() => items.length, [items])`, тоді як додавання двох чисел або читання `.length` коштує **наносекунди**. Додаткові порівняння залежностей і гілки логіки хука можуть **перевищити** економію.

- Для `useCallback` обгортати функцію, яка **передається лише в один рядок без `memo`** і не потрапляє в залежності інших хуків — тоді **немає виграшу**, але є зайвий код і зайвий порівняльний крок.

**Правило:** мемоїзувати варто те, що **реально дорого** (великі обчислення, стабільні посилання для `memo`/дочірніх `useEffect`), а не “на всяк випадок” кожен вираз.

## 5. “Why did this render?” у React Profiler, вкладка та `useCallback` зі стейтом `count`

### Що таке “Why did this render?”

Це **інструмент у React DevTools**, який показує **причину**, чому компонент **оновився** в обраному коміті: зміна **props**, **state** або **hooks** (контексту) батьків. Допомагає знайти **зайві ререндери** через нестабільні props або зайві оновлення state.

### Де знайти

У **React Developer Tools** найчастіше:

1. Вкладка **Profiler** → записати сесію (**Record**) → зупинити (**Stop**) → вибрати **коміт** на timeline → клацнути **компонент** у **Flamegraph** → у бічній панелі з’являється інформація про рендер, зокрема причина оновлення (у т.ч. блок **“Why did this render?”** або еквівалент з переліком змінених props/state — залежно від версії розширення).

2. Альтернативно в деяких версіях: вкладка **Components** → вибраний компонент → деталі останнього рендеру.

Для **аналізу непотрібних оновлень** зручно поєднати: **Profiler** (хто скільки часу зайняв) + перегляд **причини рендеру** для підозрілих компонентів.

### `useCallback` і залежність від `[count, setCount]`

Якщо всередині колбеку ви **читаєте актуальне значення** `count`, то:

```js
const handler = useCallback(() => {
  console.log(count);
}, [count]);
```

**Масив залежностей: `[count]`** (достатньо).

**`setCount` з `useState` не потрібно** додавати в залежності: `setState` від React **стабільний** за посиланням між рендерами і не змінюється.

Якщо залежності не вказати `count`, а використовувати лише `[]`, то колбек **застигне** на **старому** значенні `count` (замикання) — це типова помилка.

**Висновок:** вкажіть **`[count]`** (або `...` інші змінні, які реально читаєте всередині). Якщо використовуєте лише `setCount` без читання `count` — тоді можливий **`[]`**, бо `setCount` стабільний.
